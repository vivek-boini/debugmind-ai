# DebugMind AI — Complete System Analysis

> **STRICTLY DESCRIPTIVE**: This document explains the system as it currently exists.
> No suggestions, improvements, or redesigns are included.

---

## 1. SYSTEM OVERVIEW

### What the System Does

DebugMind AI is an agentic learning mentor system for LeetCode users. It extracts a user's submission history from LeetCode via a Chrome Extension, runs the data through a multi-agent pipeline on a Node.js/Express backend, stores results in MongoDB, and presents personalized diagnostics and learning plans through a React dashboard.

The system identifies weak topics in a user's submission history, sets target improvement goals, creates a day-by-day practice plan, monitors ongoing progress, and adjusts the learning strategy based on performance trends.

### High-Level Flow

```
User on LeetCode
      │
      ▼ (clicks "Extract My Submissions" in popup)
Chrome Extension
  contentScript.js scrapes submission table
      │
      ▼ POST /extract  { username, submissions[] }
Express Backend (port 4000)
      │
      ├─▶ Agent Orchestrator.runFullLoop()   [synchronous — blocks until done]
      │       Diagnosis → Goals → Planning → Monitoring → Adaptation
      │       Result returned in HTTP response
      │
      ├─▶ dbService.createSession()          [async — creates Session doc]
      ├─▶ dbService.persistExtractData()     [awaited — writes all DB collections]
      │
      └─▶ process.nextTick() async pipeline  [fire-and-forget, non-blocking]
              Step 0: problemService.fetchAndCacheProblems()  (LeetCode GraphQL)
              Step 1: codeAnalysisService.runLLMAnalysis()    (GROQ LLM)
              Step 2: orchestrator.runEnhancedAgentPipeline() (re-run agents with LLM data)
      │
      ▼ JSON response → Chrome Extension popup
React Dashboard (port 5173)
      │
      ├─▶ On login:  GET /load-user-data/:userId   (DB → memory hydration)
      ├─▶ On analyze: GET /agent-state/:userId      (in-memory store)
      └─▶ Renders: Dashboard, CodeAnalysis, Goals, Progress, Insights, etc.
```

## 2. DATA FLOW (STEP-BY-STEP)

### Step 1 — User Triggers Extraction

The user navigates to `https://leetcode.com/submissions/` and clicks "Extract My Submissions" in the Chrome Extension popup (`popup.js`). The popup sends a message to `contentScript.js`, which scrapes the submission table and collects fields: `title`, `titleSlug`, `status`, `statusDisplay`, `lang`, `runtime`, `memory`, `timestamp`.

### Step 2 — POST /extract

`contentScript.js` sends the collected payload to the backend:

```json
{
  "username": "leetcode_username",
  "submissions": [
    { "title": "Two Sum", "titleSlug": "two-sum", "statusDisplay": "Accepted", ... }
  ]
}
```

The backend validates the request: `username` must be a non-empty string, `submissions` must be a non-empty array.

### Step 3 — Synchronous Agent Loop (`runFullLoop`)

`orchestrator.runFullLoop(userId, submissions)` runs immediately and blocks the request until it completes. The stages are:

1. **Extract**: Submissions are added to `memoryStore` for the user.
2. **Diagnose**: `diagnosisAgent.diagnose({ submissions })` runs — classifies each submission by topic, computes per-topic success rates, identifies weak topics. Output is stored in `memoryStore`.
3. **Set Goals**: `goalAgent.setGoals(diagnosisOutput)` runs — generates SMART goals for each weak topic. Output is stored in `memoryStore`.
4. **Plan**: `planningAgent.createPlan(goalsOutput)` runs — creates a day-by-day problem practice schedule. Output is stored in `memoryStore`.
5. **Monitor**: `monitoringAgent.monitor(submissions, previousProgress, goals)` runs — computes progress metrics, trend direction, alerts. Output is stored in `memoryStore`.
6. **Adapt**: `adaptationAgent.adapt(monitoringOutput, plan, goals, diagnosis)` runs — selects a strategy modification based on rules. Output is stored in `memoryStore`.

The agent loop result (`loopResult`) is built from `memoryStore` and returned as part of the HTTP response.

### Step 4 — Session Creation (async)

`dbService.createSession(userId, submissions, metadata)` is called with `process.nextTick` effectively making it non-blocking from the agent loop's perspective, but it is `await`ed before proceeding to persistence. It creates a `Session` document in MongoDB.

### Step 5 — DB Persistence (awaited)

`dbService.persistExtractData(userId, submissions, loopResult, sessionId)` is awaited before the async LLM pipeline starts. This writes to:
- `Submission` collection — one document per unique `(userId, titleSlug)` pair, with each individual attempt stored in a subdocument array
- `AgentOutput` collection — stores the full agent loop result
- `Goal` collection — one document per goal generated by Goal Agent
- `ProgressSnapshot` collection — point-in-time snapshot of topic stats
- `Action` collection — the `nextAction` recommendation
- `Session` collection — updated with `processingStatus: 'completed'` and reference to `AgentOutput`

### Step 6 — Async LLM Pipeline (fire-and-forget)

After DB persistence completes, `process.nextTick` schedules the async pipeline:

1. **Problem Metadata Fetch**: For each unique `titleSlug` in the submitted data, `problemService.fetchAndCacheProblems(titleSlugs)` is called. This checks memory cache → MongoDB → LeetCode GraphQL API, in that order.
2. **LLM Analysis**: `codeAnalysisService.runLLMAnalysis(userId)` runs. For each of the user's 10 most recent submissions (sorted by `lastAttemptAt`), it builds a lightweight prompt and calls GROQ API. The JSON response (`mistakes`, `patterns`, `complexity`, `improvement`) is upserted into `Submission.analysis`.
3. **Enhanced Agent Pipeline**: If at least one problem was LLM-analyzed, `orchestrator.runEnhancedAgentPipeline(userId, sessionId)` re-runs agents with LLM-derived insights included. It upserts `AgentOutput` for the same session.

### Step 7 — Frontend Fetches Data

On the React dashboard:
- On login: `GET /load-user-data/:userId` is called. The backend loads MongoDB data and hydrates the in-memory `memoryStore`. The response includes `submissionDocs`, `agentOutput`, `activeGoals`, `session`.
- `AppContext` sets `agentState`, `data`, and `dataStatus`.
- Individual pages (`Dashboard`, `CodeAnalysis`, `Goals`, etc.) read from `AppContext`.

## 3. LLM USAGE

### Where LLM Is Used

LLM is used inside `codeAnalysisService.runLLMAnalysis()`, which is called as part of the async (non-blocking) pipeline during `/extract` processing.

### When It Is Triggered

LLM analysis is triggered automatically after `/extract` completes DB persistence. It does NOT run on demand and does NOT block the HTTP response. It runs in the background via `process.nextTick`.

### What Input It Receives

For each of the user's 10 most recent submission documents, the following is assembled into a prompt:

- Problem title and difficulty
- Problem description (up to 500 chars, HTML-stripped), examples, constraints, topic tags — fetched from `problemService` cache
- Total attempts count
- Accepted count and success rate
- Whether the problem is marked as solved
- Error type summary (e.g., `"Wrong Answer: 3, Time Limit Exceeded: 1"`)
- Topic tags

**No full user code is sent to the LLM.** The prompt is kept lightweight.

### What Output It Produces

The LLM (GROQ `llama-3.1-8b-instant`) is expected to return a strict JSON object:

```json
{
  "mistakes": ["likely mistake 1", "likely mistake 2"],
  "patterns": ["algorithm pattern 1", "algorithm pattern 2"],
  "complexity": "O(n) time, O(1) space",
  "improvement": "one concrete improvement tip"
}
```

### Where the Output Is Stored

The parsed JSON is upserted into the `Submission` collection via `Submission.upsertAnalysis(userId, titleSlug, analysisData)`, populating the `analysis` subdocument. The `analysis.source` field is set to `'llm'` and `analysis.lastAnalyzedAt` is set to the current timestamp.

Problems with an existing `analysis.lastAnalyzedAt` are skipped (idempotent behavior).

## 4. AGENT PIPELINE

### 4.1 Diagnosis Agent (`diagnosisAgent.js`)

**Input**: `{ submissions: [] }`

**Processing Logic**:
- Each submission's `title` is matched against `TOPIC_KEYWORDS` map (14 topic categories, each with keyword arrays) using `classifyProblem(title)`. Unmatched submissions are classified as `'General Problem Solving'`.
- Status is normalized via `normalizeStatus()` to one of: `Accepted`, `Wrong Answer`, `Time Limit Exceeded`, `Runtime Error`, `Memory Limit Exceeded`, `Compile Error`, `Other`.
- Per-topic stats are accumulated: total attempts, accepted count, failure types.
- **Success rate per topic** is computed. Topics below a 70% threshold with at least 2 submissions are marked as weak.
- Weak topics are assigned a severity: `critical` (< 30%), `high` (30–49%), `medium` (50–69%).
- A dominant error type is computed per topic (the most frequent non-Accepted status).
- **Learning velocity** is computed by splitting the timeline into two halves and comparing the success rate difference.
- **Error patterns** are aggregated across all topics (which topics are affected by `Wrong Answer`, `TLE`, etc.).
- **Confidence gaps** identify topics where the user has many attempts but low success.
- **Struggling problems** are problems with 3+ failed attempts and no accepted solution.
- A `confidence_level` is computed from overall success rate: `very_low` (< 30%), `low` (30–49%), `medium` (50–69%), `high` (70–84%), `very_high` (≥ 85%).
- `confidence_scores` is a map of topic → success rate percentage.

**Output**:
```json
{
  "weak_topics": [{ "topic": "Dynamic Programming", "score": 35, "severity": "critical", ... }],
  "error_patterns": [{ "error_type": "Wrong Answer", "affected_topics": [...], ... }],
  "confidence_gaps": [...],
  "struggling_problems": [...],
  "confidence_scores": { "Dynamic Programming": 35, ... },
  "overall_success_rate": 58,
  "confidence_level": "medium",
  "learning_velocity": { "direction": "improving", "delta": 12 },
  "total_submissions": 45,
  "total_problems": 20,
  "evidence": [...]
}
```

---

### 4.2 Goal Agent (`goalAgent.js`)

**Input**: `{ weak_topics, confidence_scores, error_patterns, confidence_level, learning_velocity, total_submissions, llm_mistakes }`

**Processing Logic**:
- For each weak topic, a `targetScore` is computed: `currentScore + target_delta`, where `target_delta` is determined by severity (`critical: 45`, `high: 35`, `medium: 25`). Target is clamped between 75 and 95. The delta is adjusted ±5 based on `learning_velocity.direction`.
- A `deadlineDays` is computed from base values (`critical: 14 days`, `high: 10`, `medium: 7`), extended for large score gaps and for declining learning velocity.
- A **goal description** is generated from `GOAL_TEMPLATES` (topic × severity matrix). If a dominant error is present, an error-specific context phrase is appended. A confidence-level phrase is also appended.
- **Milestones** are generated as equally-spaced checkpoints from `currentScore` to `targetScore` over `deadlineDays`.
- A **priority score** is computed: `(100 - currentScore) × severityWeight`.
- Goals are sorted by descending priority.
- If `llm_mistakes` are provided (from the enhanced pipeline), they are converted into additional goals with a fixed `current_score: 50`, `target_score: 75`, `deadline_days: 7`. Goals already covered by a rule-based topic are skipped.
- If no weak topics exist, the agent returns a `'no_goals_needed'` result.

**Output**:
```json
{
  "goals": [
    {
      "id": "goal_dynamic_programming_...",
      "topic": "Dynamic Programming",
      "description": "Master DP fundamentals...",
      "current_score": 35,
      "target_score": 80,
      "deadline_days": 14,
      "severity": "critical",
      "milestones": [...],
      "status": "active"
    }
  ],
  "priority_order": ["Dynamic Programming", ...],
  "summary": { "status": "...", "focus": "..." }
}
```

---

### 4.3 Planning Agent (`planningAgent.js`)

**Input**: `{ goals, diagnosis, error_patterns, confidence_level, learning_velocity }`

**Processing Logic**:
- For each goal topic (up to 3 high-priority goals), the agent selects problems from a hardcoded `PROBLEM_DATABASE` (covers 14 topics; each topic has easy/medium/hard arrays).
- Problem selection is weighted by difficulty according to severity: critical goals use 60% easy / 30% medium / 10% hard; high-severity goals use 30/50/20; medium goals use 10/60/30.
- A `problems_per_day` value is computed from confidence level (1–3 per day).
- Daily plans are generated for the duration of the longest goal deadline, typically 7–14 days. Each day has a `phase` (foundation, practice, challenge, review) and a list of practice problems.
- Phase rotation: Days 1–3 are foundation, days 4–7 are practice, days 8–11 are challenge, day 12+ is review.
- A "used problems" set prevents duplicate assignments across days.
- A `summary` string is assembled describing total days, total problems, and the primary focus topic.

**Output**:
```json
{
  "plan": [
    {
      "day": 1,
      "phase": "foundation",
      "topics": ["Dynamic Programming"],
      "problems": [{ "title": "Climbing Stairs", "difficulty": "Easy", ... }],
      "estimated_time": "45 min"
    }
  ],
  "summary": "14-day plan focusing on Dynamic Programming...",
  "total_problems": 28
}
```

---

### 4.4 Monitoring Agent (`monitoringAgent.js`)

**Input**: `{ new_submissions, previous_progress, goals, classifyProblem }`

**Processing Logic**:
- `calculateSuccessRate(submissions)` counts accepted submissions and divides by total.
- The previous success rate is taken from `previous_progress.success_rate` (or 0 if none).
- `detectTrend(currentRate, previousRate, attempts)` computes the delta. Requires a minimum of 5 submissions for reliable trend detection. Direction labels: `improving` (> 10% delta), `slightly_improving` (5–10%), `stable` (±5%), `slightly_declining` (−5 to −10%), `declining` (< −10%). For < 5 submissions: `insufficient_data`.
- A status label is assigned based on `currentRate` and `THRESHOLDS`: `IMPROVING` (≥ 60%), `STABLE` (50–60%), `STRUGGLING` (40–50%), `STAGNANT` (< 40%).
- Per-topic success rates are computed from the new submissions.
- Goal progress is checked for each active goal: percentage toward target is computed.
- A **learning velocity** is computed and an `accelerating/declining/steady` label is applied.
- Smart alerts are generated based on trend and status.

**Output**:
```json
{
  "success_rate": 62,
  "trend": { "direction": "improving", "delta": 8 },
  "status": "IMPROVING",
  "topic_progress": { "Dynamic Programming": { "success_rate": 45, ... } },
  "goal_progress": [{ "topic": "...", "percent_complete": 40 }],
  "learning_velocity": { "direction": "accelerating" },
  "alerts": [...]
}
```

---

### 4.5 Adaptation Agent (`adaptationAgent.js`)

**Input**: `{ monitoring_result, current_plan, goals, diagnosis }`

**Processing Logic**:
- `getActionFromSuccessRate(successRate)` applies `ADAPTATION_RULES` thresholds in descending order: `SUCCESS_HIGH` (≥ 80%), `SUCCESS_GOOD` (≥ 65%), `SUCCESS_MODERATE` (≥ 50%), `SUCCESS_LOW` (≥ 35%), `SUCCESS_CRITICAL` (< 35%). Returns an action label.
- Trend-based overrides are applied: a fast declining trend (delta < −15) triggers `pause_and_review`; stagnation (> 10 attempts with delta < ±5) triggers `change_strategy`; low activity (< 3 attempts) triggers `encourage_activity`.
- `STRATEGY_MODIFICATIONS` map maps action label to `{ difficulty_adjustment, problems_per_day_adjustment, focus_shift, message }`.
- Topic-specific recommendations are generated for the weakest topic.
- A `next_steps` list is assembled from the monitoring alerts and goal status.

**Output**:
```json
{
  "action": "add_practice",
  "reason": "Moderate performance - add more practice problems",
  "adjustments": { "difficulty_adjustment": 0, "problems_per_day_adjustment": 1 },
  "recommendations": ["Focus on Dynamic Programming fundamentals"],
  "next_steps": [...]
}
```

## 5. PROBLEM SERVICE

### How Problem Data Is Fetched

`problemService.js` fetches problem metadata from the LeetCode GraphQL endpoint (`https://leetcode.com/graphql`). It sends a POST request with the query `getProblem(titleSlug)`, requesting: `title`, `content`, `difficulty`, `topicTags { name slug }`, `exampleTestcases`, `similarQuestions`. The timeout is 8 seconds.

### What Fields Are Extracted

- `title` — problem display name
- `shortDescription` — HTML-stripped `content`, trimmed to `MAX_DESCRIPTION_LENGTH` (500 chars)
- `difficulty` — `Easy` / `Medium` / `Hard`
- `tags` — normalized tag names (via `TAG_NORMALIZE_MAP`; falls back to inference if empty)
- `examples` — raw `exampleTestcases` string (max 500 chars)
- `constraints` — extracted via regex match on `Constraints:` section (max 300 chars)
- `similarProblems` — parsed from `similarQuestions` JSON string (up to 5 items: `title`, `titleSlug`, `difficulty`)
- `fetchedAt` — timestamp

### How Caching Works

The cache is two-layered:

**Memory cache**: A `Map` (`memoryCache`) keyed by normalized `titleSlug`. Entries expire after 24 hours (`CACHE_TTL`). On a hit, data is returned immediately without DB or API access.

**DB cache**: If the memory cache misses, `Problem.findOne({ titleSlug })` is queried in MongoDB. On a hit, the result is also written to the memory cache and returned.

### How Fallback Logic Works

`getProblemWithCache(titleSlug)` implements the three-layer waterfall: memory → DB → API. On any layer failure it falls through to the next. If all three fail, `null` is returned — the function never throws.

**Tag fallback** (three layers):
1. Primary: `topicTags` from LeetCode API, normalized through `TAG_NORMALIZE_MAP`
2. Secondary (if empty): `inferTagsFromContent(title, description)` — keyword matching across `KEYWORD_TAG_MAP` (20 topics), returning up to 5 matched tags
3. Final fallback: `['General']`

**similarProblems fallback**: If `JSON.parse(similarQuestions)` fails, `similarProblems` defaults to `[]`.

**Rate limiting**: A 200ms delay is inserted between API calls when the memory cache does not already contain the slug. The batch size is capped at `MAX_BATCH_SIZE = 10`.

## 6. DATABASE STRUCTURE

### `users` collection

**Schema** (via `userSchema`):
- `userId`: lowercase LeetCode username (unique, primary key)
- `email`: optional, unique (sparse index)
- `password`: bcrypt-hashed, excluded from default queries (`select: false`)
- `leetcodeUsername`: lowercase
- `displayName`: optional
- `authType`: `'password'` | `'guest'`
- `isGuest`: boolean
- `preferences`: `{ dailyGoal: Number, preferredTopics: [String], difficulty: String }`
- `stats`: `{ totalSessions, totalSubmissions, currentStreak, longestStreak, lastExtractDate }`
- `lastSessionId`: reference to most recent session
- Timestamps: `createdAt`, `updatedAt`

**When written**: On `/signup` (full record) or `/guest-login` (minimal guest record via `findOrCreateGuest`). Password is hashed pre-save if set.

**How used later**: User lookup on `/login` and `/me`. `findExistingUser` checks for duplicate email or username during signup.

---

### `submissions` collection

**Schema** (via `submissionSchema`):
- Unique key: `(userId, titleSlug)` — one document per problem per user
- `title`, `difficulty`, `topics`, `topicTags`
- `submissions[]`: array of `submissionEntrySchema` — each individual attempt with `code`, `runtime`, `memory`, `status`, `language`, `sessionId`, `timestamp`
- `stats`: `{ totalAttempts, acceptedCount, wrongAnswerCount, tleCount, firstAttemptAt, lastAttemptAt, firstAcceptedAt, bestRuntime, bestMemory, isSolved, attemptsToSolve }`
- `analysis`: `{ source, mistakes, patterns, complexity, improvement, lastAnalyzedAt }`

**When written**: `Submission.addSubmission()` is called for each incoming submission during `persistExtractData`. It upserts by `(userId, titleSlug)`, pushing new attempts. `stats` are recomputed on every save. `analysis` is populated later by the async LLM pipeline via `Submission.upsertAnalysis()`.

**How used later**: Loaded by `dbService.getAllUserSubmissions()` for LLM analysis and for `/load-user-data` hydration. `submissionDocs` are sent to the frontend as part of `CodeAnalysis` page data.

---

### `problems` collection

**Schema** (via `problemSchema`):
- `titleSlug`: unique primary key
- `title`, `shortDescription`, `difficulty`
- `tags`: normalized tag names
- `examples`, `constraints`
- `similarProblems[]`: `{ title, titleSlug, difficulty }`
- `fetchedAt`: when last fetched from LeetCode

**When written**: Written (upserted) by `getProblemWithCache()` after a successful LeetCode API fetch. Only written once per problem unless manually cleared.

**How used later**: Read by `getProblemWithCache()` as the DB cache layer. Also read by `codeAnalysisService` when building LLM prompts.

### `sessions` collection

**Schema** (via `sessionSchema`):
- `sessionId`: UUID (unique)
- `userId`
- `rawExtractedData`: `{ submissions[], metadata: { extractionTimestamp, source, extensionVersion } }`
- `summaryStats`: auto-computed pre-save from raw submissions — `{ totalSubmissions, acceptedCount, uniqueProblems, successRate, languages, difficulties }`
- `analysisResults`: weak/strong topics, overall success rate
- `processingStatus`: `'pending'` → `'processing'` → `'completed'` | `'failed'`
- `agentOutputId`: ObjectId reference to `AgentOutput`
- `agentVersion`: `'2.2.0'`

**When written**: Created by `dbService.createSession()` at the start of `/extract`. Updated with `processingStatus` and `agentOutputId` after pipeline completes.

**How used later**: Loaded by `dbService.loadUserData()` to find the most recent session. `summaryStats` is sent to the frontend.

### `agentoutputs` collection

**Schema** (via `agentOutputSchema`):
- `userId`, `sessionId` (unique together — upserted)
- `diagnosis`, `goals[]`, `plan`, `monitoring`, `adaptation` — all `Mixed` type
- `llmSummary`: `{ commonMistakes, weakPatterns, improvementAreas, analyzedCount, lastAnalyzedAt }`
- `decisions[]`: agent decision log entries
- `processingTime`
- `status`: `'pending'` | `'completed'` | `'failed'`

**When written**: First written by `dbService.persistExtractData()` after the synchronous agent loop. Then **upserted again** (same `userId + sessionId`) by `orchestrator.runEnhancedAgentPipeline()` after the LLM analysis completes. The second write includes `llmSummary` populated from analyzed submissions.

**How used later**: Loaded by `dbService.loadUserData()` and sent to frontend as `agentOutput`. Used to hydrate `memoryStore` on `/load-user-data`. Contains the full `diagnosis`, `goals`, `plan`, `monitoring`, `adaptation` that the dashboard renders.

### `goals` collection

**Schema** (via `goalSchema`):
- `userId`, `goalId` (UUID, unique)
- `sessionId`
- `title`: e.g., `"Improve Dynamic Programming"`
- `description`: e.g., `"Increase Dynamic Programming success rate from 35% to 80%"`
- `topic`
- `targetMetric`: `{ type: 'success_rate', value: 80, unit: 'percent' }`
- `currentMetric`: `{ value: 35, lastUpdated: Date }`
- `status`: `'active'` | `'completed'` | `'paused'` | `'abandoned'`
- `priority`: number
- `progress[]`: historical value snapshots
- `dueDate`, `completedAt`

**When written**: Created by `Goal.createFromAgentGoals()` inside `persistExtractData`, once per goal in the agent output.

**How used later**: Loaded by `dbService.loadUserData()` as `activeGoals` (filtered to `status: 'active'`). Used to populate `Goals` page in the frontend. Also used to hydrate `memoryStore` if `agentOutput.goals` is empty.

---

### `progresssnapshots` collection

**Schema** (via `progressSnapshotSchema`):
- `userId`, `sessionId`
- `topicStats`: Map of topic → `{ attempts, accepted, successRate, avgAttempts }`
- `overallStats`: `{ successRate, totalAttempts, totalAccepted, uniqueProblems, solvedProblems }`
- `difficultyBreakdown`: easy/medium/hard stats
- `streakInfo`: `{ currentStreak, longestStreak, lastActiveDate }`
- `comparisonToPrevious`: `{ successRateChange, newProblemsSolved, improvementScore }`

**When written**: Created by `ProgressSnapshot.createSnapshot()` inside `persistExtractData`. The previous snapshot is fetched (by `userId`, sorted by `createdAt` descending) to compute `comparisonToPrevious`.

**How used later**: Not currently loaded back into the frontend. Used for historical trend comparison when creating future snapshots.

---

### `actions` collection

**Schema** (via `actionSchema`):
- `userId`, `sessionId`
- `actionType`: one of a fixed enum (e.g., `'practice_problems'`, `'review_topic'`, `'focus_weak_area'`, etc.)
- `action`: the primary recommended action text
- `description`, `reason`
- `suggestedProblems[]`: `{ title, titleSlug, difficulty, topic, url }`
- `priority`: number (1–4)
- `status`: `'pending'` | `'in_progress'` | `'completed'` | `'ignored'` | `'expired'`
- `expiresAt`: 7 days from creation

**When written**: Created by `Action.createFromNextAction()` inside `persistExtractData` from the `nextAction` object generated by `nextActionGenerator.generateNextAction()`.

**How used later**: Loaded by `dbService.loadUserData()`. Referenced in `agentOutput.nextAction` sent to the frontend.

## 7. BACKEND FLOW

### /extract Flow

1. Request body validated: `username` (string), `submissions` (non-empty array).
2. `username` is lowercased and trimmed.
3. `dbService.createSession()` is invoked — returns a Promise stored in `sessionPromise`.
4. `orchestrator.runFullLoop(userId, submissions)` is called — **synchronous block** on the request.
5. `sessionPromise` is awaited.
6. `dbService.persistExtractData()` is **awaited** — all DB writes complete before the async pipeline starts.
7. `process.nextTick()` schedules the async pipeline (problem fetch → LLM → enhanced agents). This does not block the response.
8. HTTP 200 JSON response is sent with `{ status, user_id, received, agent_loop, next_action, alerts, session_id }`.

### Async Processing Behavior

The synchronous part (steps 1–8) completes in one event loop cycle. The async pipeline (LLM + enhanced agents) runs in a subsequent tick and has no mechanism to push results to the already-sent HTTP response. The frontend must re-fetch to see LLM-enhanced results.

The frontend handles this via a 10-second delayed re-fetch after login, and polling every 3 seconds (up to a 30-second safety timeout) after analyzing.

### DB Persistence Flow

`persistExtractData` performs writes in the following order, mixing awaited and background operations:

**Awaited (blocks until complete — LLM pipeline depends on these being done):**
1. `updateSessionStatus(sessionId, 'processing')` — marks session as in-progress
2. `storeAgentOutput(userId, sessionId, loopResult)` — upserts `AgentOutput` document (`AgentOutput.createFromLoopResult`)
3. `linkAgentOutputToSession(sessionId, agentOutput._id)` — updates `Session.agentOutputId`
4. `updateSessionAnalysis(sessionId, diagnosis)` — updates `Session.analysisResults` and sets `processingStatus: 'completed'`
5. `storeSubmissionsAwaited(userId, submissions, sessionId)` — upserts each submission via `Submission.bulkAddSubmissions`

**Background (non-blocking — `setImmediate` via `backgroundOperation`):**
6. `createProgressSnapshotBackground` — creates `ProgressSnapshot` document
7. `storeGoalsWithLogging` — creates `Goal` documents via `Goal.createFromAgentGoals`
8. `storeActionWithLogging` — creates `Action` document via `Action.createFromNextAction`
9. `updateUserStats` — updates `User.stats.totalSubmissions` and `User.lastSessionId`

All writes run inside `dbService.safeOperation()` wrappers, which return `null` on any failure instead of throwing.

### Logging Behavior

The backend uses `console.log` / `console.warn` / `console.error` with bracketed prefixes (e.g., `[Extract]`, `[AsyncPipeline]`, `[DBService]`, `[ProblemService]`, `[LLMService]`).

Request logging middleware logs method, path, status code, and duration for every request. Requests with status ≥ 400 use `console.error`.

Agent decision logging uses `agentLogger.logDecision()`, which stores structured decision objects in a separate in-memory `logStore` (keyed by `userId`). Logs include: `agent`, `decision`, `reason`, `confidence`, `input_summary`, `output_summary`, `evidence[]`, `human_readable`. These are accessible via `GET /agent-logs/:userId`.

## 8. FRONTEND FLOW

### How Data Is Fetched

`AppContext.jsx` is the central state manager. It exposes `data`, `agentState`, `dataStatus`, `loading`, and all action functions via React Context.

On mount, it checks `localStorage` for cached `debugmind_data` and pre-fills `data` if present.

On login:
1. `POST /login` or `POST /guest-login` is called.
2. `loadUserDataFromDB(userId)` calls `GET /load-user-data/:userId`.
3. If the response has `hasData: true`, `agentState` and `data` are set from the response.
4. 10 seconds after login, `loadUserDataFromDB` is called again to pick up LLM-enhanced results.

On `analyze()`:
1. `POST /analyze` is called with the user's profile URL.
2. `isPolling` is set to `true`.
3. A polling loop runs every 3 seconds calling `GET /agent-state/:userId`, stopping when `status === 'ready'` and `submissions_count > 0`, or after 30 seconds.

### How the CodeAnalysis Page Works

`CodeAnalysis.jsx` reads `agentState` and `data.submissionDocs` from `AppContext`. It does not make its own API calls.

`submissionDocs` is the array of `Submission` MongoDB documents. Each document has `title`, `titleSlug`, `difficulty`, `topics`, `submissions[]`, `stats`, and `analysis` fields.

The page renders:
- A summary header with total problems, solved count, and overall success rate.
- Cards for each submission document showing difficulty badge, topic tags, attempt count, success rate, and best runtime.
- If `analysis.lastAnalyzedAt` exists on a document (i.e., LLM analysis ran), the `analysis` fields (`mistakes`, `patterns`, `complexity`, `improvement`) are displayed.
- A `parseAnalysis()` utility function parses plain-text LLM output as a fallback if structured fields aren't present.

### How ProblemDetailsCard Is Rendered

Each submission doc is rendered as an expandable card. On expansion, the full submission history (status, language, runtime, memory, timestamp) is shown. If `analysis` data is present, the card renders: identified mistakes (as a bulleted list), algorithmic patterns, complexity estimate, and an improvement tip.

### What Comes from DB vs Computed in UI

**From DB** (`/load-user-data` response):
- `submissionDocs`: full submission history with pre-computed `stats` and `analysis`
- `agentOutput`: `diagnosis`, `goals[]`, `plan`, `monitoring`, `adaptation`
- `session.summaryStats`: overall counts (totalSubmissions, acceptedCount, etc.)
- `activeGoals`: active goal documents from the `goals` collection

**Computed in UI** (inside `AppContext` or page components):
- `weak_topics` array — derived in `AppContext` from `agentOutput.goals` (each goal's `topic`, `current_score`, `target_score`)
- `hasData` flag — computed from `dataStatus === 'ready' && !!data && !!agentState?.submissions_count`
- Score badge colors, difficulty badge colors, status chip colors — computed by `getScoreColor()`, `getDifficultyColor()`, `getStatusColor()` in `CodeAnalysis.jsx`
- Progress bar widths and percentages — computed from `current_score` / `target_score` ratios

## 9. CACHING STRATEGY

### What Is Cached

1. **LeetCode problem metadata** (title, description, tags, difficulty, constraints, similarProblems)
2. **User agent state** (submissions, diagnosis, goals, plan, progress, adaptation) — in-memory via `memoryStore`
3. **Auth data** (token, user object) — in browser `localStorage`
4. **Dashboard data** — in browser `localStorage` as `debugmind_data`

### Where (Memory vs DB)

| Data | Memory Cache | DB Cache |
|------|-------------|----------|
| Problem metadata | `Map` in `problemService.js` (24h TTL) | `problems` collection |
| Agent state / submissions | `Map` in `memoryStore.js` (no TTL, process lifetime) | `sessions`, `submissions`, `agentoutputs` |
| Auth token | `localStorage` (browser) | N/A |
| Dashboard data | `localStorage` (browser) | N/A |

### When Cache Is Used vs API Call

**Problem metadata**:
- Memory cache hit → returned immediately, no DB or API call
- Memory cache miss, DB hit → returned, also written to memory cache
- Both miss → LeetCode GraphQL API is called, result written to both memory and DB

**Agent state**:
- `GET /agent-state/:userId` returns from `memoryStore` (in-memory only)
- `memoryStore` is populated: (a) when new data is extracted via `/extract`, or (b) when `GET /load-user-data/:userId` is called and hydrates from DB

**Browser caches**:
- On mount, `AppContext` reads `localStorage` for `debugmind_data` and pre-fills state without any API call
- Auth state is re-read from `localStorage` on every page load via `useState` initializer

## 10. ERROR HANDLING

### When LLM Fails

`generateLLMInsights()` in `llmService.js` catches all Axios errors and returns `{ success: false, text: null, error: errMsg }` — it never throws. If the GROQ API key is absent, it returns immediately with `error: 'API key not configured'`.

In `codeAnalysisService.runLLMAnalysis()`, each problem is analyzed inside a `try/catch`. On failure:
- `result.errors` is incremented
- Processing continues to the next problem

If `tryParseJSON(llmResult.text)` fails to parse the LLM response, the error count is incremented and the problem is skipped.

If LLM analysis produces zero results (`llmResult.analyzed === 0`), the enhanced agent pipeline is not triggered.

### When Problem Fetch Fails

`fetchProblemDetails()` catches all Axios errors. Rate limit (HTTP 429) is logged as a warning. All other errors are logged and `null` is returned.

`getProblemWithCache()` falls through silently if any layer fails.

In `codeAnalysisService.runLLMAnalysis()`, the problem context fetch is wrapped in its own `try/catch`. If it fails, `problemContext` is set to an empty string and analysis continues with title-only context.

In the async pipeline in `/extract`, the entire problem fetch block is wrapped in `try/catch` with a non-fatal warning log.

### When DB Fails

All MongoDB operations go through `dbService.safeOperation()`, which:
1. Checks `isDBConnected()` before attempting — returns `null` if not connected
2. Wraps the operation in `try/catch` — logs the error and returns `null` on failure

If MongoDB is unavailable at startup, the server logs `"Running without MongoDB - memory-only mode"` and continues operating with in-memory state only. The `isConnected` flag stays `false`.

DB-write failures during `/extract` are logged but do not cause the HTTP response to fail. The agent loop result is still returned.

Session creation failure is caught and logged — the variable `sessionId` remains `null`, and the rest of the flow continues without a session reference.

## 11. SYSTEM BEHAVIOR

### Sync vs Async Operations

**Synchronous (blocking the HTTP response)**:
- Input validation
- `orchestrator.runFullLoop()` — all 5 agent computations
- `dbService.createSession()` — awaited
- `dbService.persistExtractData()` — awaited

**Asynchronous (fire-and-forget, non-blocking)**:
- `problemService.fetchAndCacheProblems()` — LeetCode GraphQL calls
- `codeAnalysisService.runLLMAnalysis()` — GROQ API calls
- `orchestrator.runEnhancedAgentPipeline()` — agent re-run with LLM data

The async pipeline is initiated via `process.nextTick()` after the response-blocking section completes. Its results only appear in subsequent requests.

### What Blocks Request vs What Runs in Background

| Operation | Blocks /extract response? |
|-----------|--------------------------|
| Agent loop (5 agents) | Yes |
| Session creation | Yes (awaited before response) |
| DB persistence (submissions, agentoutputs, goals, snapshots, actions) | Yes (awaited before async pipeline) |
| Problem metadata fetch (LeetCode GraphQL) | No (async pipeline) |
| LLM analysis (GROQ) | No (async pipeline) |
| Enhanced agent re-run | No (async pipeline) |

### How UI Updates

The frontend does not maintain a WebSocket or SSE connection. Updates happen through polling:

1. On extraction: the extension receives the initial `/extract` response and displays the synchronous agent loop result.
2. On dashboard load: `loadUserDataFromDB` is called, which returns DB data (synchronous agent results initially).
3. 10 seconds after login: `loadUserDataFromDB` is called again to pick up any LLM-enhanced data.
4. When `isPolling` is `true`: `fetchAgentState` is called every 3 seconds, stopping when `status === 'ready'` or after a 30-second safety timeout.
5. The in-memory `memoryStore` is the data source for `GET /agent-state/:userId`. It is hydrated from DB by `GET /load-user-data/:userId`.

## 12. CURRENT LIMITATIONS (ONLY OBSERVATION)

### No Real-Time Push

The frontend has no WebSocket or server-sent events connection. The async LLM pipeline results can only be seen after a separate polling round or manual re-fetch. The 10-second delayed re-fetch is a fixed timer, not triggered by actual pipeline completion.

### In-Memory Store Is Not Persistent Across Restarts

`memoryStore.js` uses a JavaScript `Map`. When the Node.js process restarts, all in-memory state is lost. Users must call `/load-user-data` to re-hydrate the store from MongoDB.

### LLM Analysis Is Not Triggered Without DB

`codeAnalysisService.runLLMAnalysis()` fetches submission docs from `dbService.getAllUserSubmissions()`. If MongoDB is unavailable, no LLM analysis runs — the function returns `{ analyzed: 0, skipped: 0, errors: 0 }` silently.

### Problem Classification Is Title-Based Only (Initial)

`diagnosisAgent.classifyProblem(title)` matches against title keywords. Problems whose titles don't match any keyword are classified as `'General Problem Solving'`. The richer tag data from `problemService` (fetched after diagnosis) is not fed back into the diagnosis agent's initial classification.

### Diagnosis and Goal Agents Run Before LLM Data Is Available

The synchronous `runFullLoop()` runs before problem metadata is fetched and before LLM analysis is complete. The initial diagnosis is based solely on submission status and title-keyword matching. LLM-enhanced data (`llm_mistakes`) reaches the Goal Agent only during the second, asynchronous `runEnhancedAgentPipeline()` call.

### Problem Database Is Hardcoded

The `PROBLEM_DATABASE` in `planningAgent.js` is a static, hardcoded list of LeetCode problems organized by topic and difficulty. It is not synchronized with MongoDB or the LeetCode API.

### No Deduplication of Repeated LLM Analysis

The `analysis.lastAnalyzedAt` check in `codeAnalysisService` prevents re-analyzing already-analyzed problems. However, if the same submission is extracted again in a new session, the existing analysis is preserved due to the `upsert: false` in `Submission.upsertAnalysis()` — but the deduplication check reads from `doc.analysis.lastAnalyzedAt` which would already be set.

### `codeAnalysis` State in Frontend Is Deprecated

`AppContext` maintains a `codeAnalysis` state variable that is never actively populated. `fetchCodeAnalysis` returns `null` immediately with a deprecation warning. LLM analysis data is instead accessed through `data.submissionDocs[].analysis`.

### Session Data vs Submission Data Divergence

The `rawExtractedData.submissions` array stored in `Session` is the raw extension payload. The normalized and deduplicated data lives in the `Submission` collection. There is no direct foreign key linking individual `Session.rawExtractedData.submissions` entries to specific `Submission` documents.

### No Streaming or Incremental UI Updates

The frontend `Progress` page and `Dashboard` read from `agentState` at a single point in time. Ongoing agent pipeline execution is not reflected in real-time — only the final completed state is surfaced.

### GROQ API Dependency

The LLM analysis pipeline depends entirely on the GROQ API (`llama-3.1-8b-instant` model). If `GROQ_API_KEY` is not set in the environment, all LLM-derived features (code analysis, enhanced goals) are silently skipped. No fallback LLM or local model is used.

### User Stats Not Auto-Updated

The `User.stats` fields (`totalSessions`, `totalSubmissions`, `currentStreak`, `longestStreak`) are updated during `persistExtractData` but `currentStreak` and `longestStreak` are incremented by fixed values and not computed from actual date-based activity. `lastExtractDate` is not set in the current implementation.
